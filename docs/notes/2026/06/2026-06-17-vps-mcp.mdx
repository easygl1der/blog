## Summary

我现在的做法，不是把 MCP 看成一两个“插件”，而是把它看成一整层 AI 工具基础设施。香港 VPS 负责跑这些工具，域名负责把它们稳定暴露给 Agent，OpenKB 负责搜索主导，LaTeX 编译、语言编译、笔记写入和其他专用能力则分别作为独立工具挂上去。

真正重要的不是“我接了多少个 MCP”，而是这些 MCP 是否形成了一个清晰的分层：搜索优先、专用工具次之、结果可验证、工作流可重复。

---

## What This Solves

我最想解决的是一个很现实的问题：当 AI 能力越来越多时，真正难的反而不是模型本身，而是怎么把外部能力组织成一套稳定、可调用、可扩展的工具系统。

过去很多人理解 MCP，会把它想成“给模型接一个接口”。这个理解不算错，但太小了。现在更合适的理解是：MCP 是 AI 的工具总线。你可以把搜索、检索、编译、写作、发布、验证、抓取、转换这些能力拆成独立模块，再让 Agent 通过统一协议去调用。

我在香港的 VPS 上做的事情，就是把这条总线自己搭起来。不是为了炫技，而是为了让 AI 真正进入专业场景：

- 想找知识时，先走 OpenKB 主导的搜索
- 想写笔记时，走专门的笔记/发布链路
- 想处理 LaTeX 时，走编译工具
- 想处理某种语言或构建任务时，走对应的编译工具
- 想让 Agent 做复合工作时，把这些工具串成一条工作流

<Info>
这篇文章的核心不是“我用了什么黑科技”，而是“我怎样把 AI 需要的能力拆开、命名、暴露、验证，再让 Agent 稳定地调用它们”。
</Info>

## Who This Is For

这篇适合已经开始自建 AI 工具的人，尤其是下面几类人：

- 想把个人知识库做成可搜索、可复用、可增量更新系统的人
- 想把编译、转换、写作、发布这类固定工作交给 Agent 的人
- 手上已经有 VPS、域名、Docker 或类似基础设施的人
- 希望 AI 不是只会聊天，而是真的能做事的人

如果你只是想“装一个 MCP 试试看”，这篇会显得偏重。如果你已经开始在意可靠性、命名、工具分层、搜索优先级和长期维护，这篇会更接近你要的东西。

## Prerequisites

- 一台长期在线的 VPS，最好是你能稳定控制的私有机器
- 一个可以对外访问的域名，或者至少是稳定的 HTTPS 入口
- 一套可维护的 MCP 服务部署方式
- 一个 Agent 客户端，能连接自定义 MCP 服务
- 至少一个稳定的知识源，比如笔记、博客、文档或资料库
- 至少一个可验证的专用工具，比如编译器、检索器或转换器

<Warning>
不要先从“做一个很大的智能平台”开始。先把一个工具做通、做稳、做可验证，再扩展到下一层。
</Warning>

## The Workflow

<Steps>
  <Step title="先把 MCP 当成工具分层，而不是功能堆叠">
    我现在会先问自己：这个能力属于哪一层。搜索是一层，编译是一层，写入和发布是一层，浏览器操作是一层，数据转换又是一层。分层以后，Agent 调用就会清楚很多，也更容易替换实现。
  </Step>
  <Step title="把 OpenKB 放在搜索主导位">
    在我现在的系统里，最重要的入口不是“让模型随便问”，而是“先让它搜”。OpenKB 适合当主搜索层，因为它对个人知识、笔记、长文、索引和语义检索更友好。Agent 先在 OpenKB 里找答案，再决定是否需要调用别的工具。
  </Step>
  <Step title="把专用能力拆成独立 MCP 服务">
    例如 LaTeX 编译、语言编译、文档转换、笔记写入、网页抓取、结果验证，都可以拆成独立服务。这样做的好处是，每个工具都有清楚的输入输出，失败时也更容易定位。
  </Step>
  <Step title="用域名把这些服务统一暴露出来">
    我会用一个稳定的域名入口把服务挂起来，而不是让 Agent 记住一堆零散地址。这样后续换端口、换容器、换实例，外部调用方式都可以保持不变。
  </Step>
  <Step title="让 Agent 先搜索，再执行，再验证">
    一个好的工作流通常不是“直接执行”，而是“先搜索确认，再调用工具，再检查结果”。比如先用 OpenKB 找到相关笔记或旧决策，再去编译或写入，最后检查输出是否和预期一致。
  </Step>
  <Step title="把复合任务封装成可重复的路由">
    当某个工作反复出现，就不再把它当临时手工操作，而是做成稳定路由。比如“查资料 -> 生成摘要 -> 写笔记 -> 发布”可以是一条路由，“找知识 -> 编译 LaTeX -> 修正错误”可以是另一条路由。
  </Step>
</Steps>

### 我现在会暴露出来的能力类型

我会把能力大致分成几组，而不是混成一锅：

1. **知识搜索类**：OpenKB 主导搜索、语义检索、历史笔记召回、专题查找。
2. **内容生产类**：写笔记、生成草稿、整理卡片、把原始想法转成结构化文稿。
3. **编译类**：LaTeX 编译、其他语言/构建编译、错误回传、产物检查。
4. **转换类**：PDF、文档、结构化文本、图片或页面内容的抽取与变换。
5. **验证类**：检查输出是否成功、编译是否通过、结果是否可读。
6. **浏览器类**：需要真实网页操作时，再接浏览器自动化工具。
7. **工作流类**：把多个原子工具串起来，形成一条更大的任务链。

我现在更倾向于这种思路：**让工具自己做小事，让 Agent 负责组合小事**。这样既灵活，也不会把复杂度全部塞进一个超大 MCP 服务里。

<Info>
我已经实际验证过 OpenKB 服务是活的，并且可以通过文本搜索拿到结果。它的能力面至少包括文本搜索、URL 入库、文本入库、健康检查，以及图像资产检索这几类。
</Info>

### 一个更像样的搜索策略

如果我要给这套系统定义一个默认策略，我会这么写：

- 第一优先级：OpenKB 搜索
- 第二优先级：项目内文档和笔记
- 第三优先级：专用工具执行结果
- 第四优先级：浏览器或外部网页确认

这样做的原因很简单。很多 AI 误判都不是“不会做”，而是“先做了再说”。搜索前置，能显著减少无谓执行，也能把已有知识重新利用起来。

```text
用户问题
  -> OpenKB 搜索
  -> 命中笔记/概念/旧决策
  -> 判断是否需要编译、转换或发布
  -> 调用专用 MCP 工具
  -> 检查输出
  -> 返回结果
```

### 我会怎么给工具命名

命名最好不要太泛。

- `search` 太泛，最好细化成 `openkb_search`
- `compile` 太泛，最好细化成 `latex_compile` 或某个语言专用编译器
- `write` 太泛，最好细化成 `note_write` 或 `publish_note`
- `convert` 太泛，最好明确输入输出类型

我现在更在意的是：**名字要让 Agent 一眼知道这工具属于哪一层、做什么、什么时候该用。**

## Common Failure Modes

<Warning>
把 MCP 当成“一个万能入口”会很快变乱。结果通常是：搜索、编译、发布、抓取、验证全挤在一起，任何一个地方出错都不好排查。
</Warning>

<Warning>
只做“能连上”，不做“能验证”，会让工具看起来很完整，实际上不可用。一个真正可用的 MCP，应该能跑一个小的真实调用，看见真实结果。
</Warning>

<Warning>
把 OpenKB、LaTeX、笔记系统和浏览器操作混成一条链，会让系统失去边界。搜索应该负责找，编译应该负责算，写入应该负责存，浏览器应该负责 UI。
</Warning>

<Warning>
如果你把每个工具都暴露成同样宽的接口，Agent 反而更容易乱用。专业场景里，清晰边界比“什么都能做”更重要。
</Warning>

<Tip>
先验证一个最小闭环：搜索能找到东西，编译能产出结果，写入能保存内容，Agent 能读懂工具返回。闭环成立以后，再扩展能力。
</Tip>

## Final Checklist

- [ ] OpenKB 是默认搜索入口，而不是可选附属功能
- [ ] 每个专用能力都有清晰的 MCP 工具边界
- [ ] Agent 先搜索，再执行，再验证
- [ ] 域名和服务入口是稳定的，不依赖临时地址
- [ ] 至少跑通过一次真实调用，而不是只看配置
- [ ] 失败时能回传明确错误，而不是只返回空结果
- [ ] 能够把一条复合任务拆成可重复的工作流

## What To Remember

MCP 最有价值的地方，不是“多接几个工具”，而是把 AI 的能力外接成一个真正可维护的系统。

如果说过去的思路是“让模型什么都会一点”，现在更好的思路是“让模型知道什么时候调用什么工具”。在这个意义上，香港 VPS 不是单纯的服务器，而是我给 AI 搭的一层外部工作台：OpenKB 负责找，编译器负责算，笔记系统负责存，Agent 负责编排。

这就是我现在理解的 MCP：它不是一个插件盒子，而是一套可以被你重新设计的 AI 工具层。
