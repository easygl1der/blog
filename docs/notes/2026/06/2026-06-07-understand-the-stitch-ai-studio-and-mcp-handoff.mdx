## Summary

The useful mental model is that Stitch is the design and interaction-context layer. It creates screens, visual systems, HTML/CSS, screenshots, and design metadata; AI Studio, Antigravity, Codex, Claude Code, Cursor, or Gemini CLI can then use that context to generate or continue an app.

The subtle relationship is not a magic full-stack bridge. It is a handoff system: Stitch makes the product interface legible to an agent, and the receiving agent still has to decide data models, auth, APIs, storage, deployment, and production boundaries.

---

## What This Solves

It is tempting to think of Stitch and AI Studio as one hidden product pipeline:

```text
Stitch design -> AI Studio -> complete full-stack website
```

That can be true for a quick prototype, but it is not the most accurate mental model. Stitch's durable contribution is not backend architecture; it is structured design context.

Stitch is closer to:

```text
Stitch
  -> screens
  -> flow
  -> design theme
  -> HTML/CSS
  -> screenshots
  -> DESIGN.md
  -> export or MCP payload
```

The receiver then turns that payload into code or a prototype.

<Info>
The important distinction is between visible UI context and application architecture. Stitch can describe the former. A coding agent still has to build the latter.
</Info>

## Who This Is For

This is for builders using Stitch and wondering whether AI Studio has a special privileged path that makes the app full-stack automatically.

It is also for anyone deciding whether to send a Stitch design into AI Studio, Antigravity, Codex, Claude Code, Cursor, Gemini CLI, or a local Next.js project.

## Prerequisites

- A Stitch project or screen with enough UI detail to represent the product.
- A target receiving environment, such as AI Studio, Antigravity, Codex, Claude Code, Cursor, or Gemini CLI.
- A stack decision if the goal is a real app, such as Next.js, Supabase or Firebase, GitHub, and Vercel.
- A willingness to separate design handoff from backend implementation.

## The Workflow

<Steps>
  <Step title="Treat Stitch as the design context source">
    Use Stitch to settle the screens, visual style, navigation, user flow, and component intent. Do not expect it to infer secure backend behavior from the visuals alone.
  </Step>
  <Step title="Choose the receiving agent">
    Use AI Studio when you want a Google/Gemini-centered prototype or shareable app path. Use Antigravity, Codex, Claude Code, Cursor, or Gemini CLI when you want the design to enter an engineering workflow.
  </Step>
  <Step title="Export or read the Stitch screen">
    Use Stitch exports, the Stitch MCP server, or Stitch skills to retrieve the screen data. The useful artifacts are usually HTML code, screenshot URL, screen dimensions, device type, design theme, and project metadata.
  </Step>
  <Step title="Create a design system document">
    Generate a `DESIGN.md` from the Stitch project when the design should be reused across multiple pages or by multiple agents. This makes visual decisions explicit instead of hidden inside one screen.
  </Step>
  <Step title="Ask for an implementation brief before code">
    Have the receiving agent extract screens, components, user flows, implied data models, backend requirements, auth rules, storage needs, API boundaries, and build phases.
  </Step>
  <Step title="Build frontend first">
    Convert the Stitch screens into React or Next.js components with mock data. This checks whether the handoff preserved the UI before backend complexity is added.
  </Step>
  <Step title="Add backend intentionally">
    Add Supabase or Firebase, server-side API routes, protected AI calls, storage, and deployment only after the UI and data model are clear.
  </Step>
</Steps>

A practical prompt for the receiving agent:

```text
Read this Stitch project through Stitch MCP.
Do not write code yet.

Extract:
1. screens and user flows
2. reusable components
3. visual tokens and design rules
4. implied data models
5. backend requirements
6. auth, storage, and API requirements
7. implementation phases

Assume the target stack is Next.js, Supabase, GitHub, and Vercel.
```

Then start implementation separately:

```text
Implement phase 1 only:
convert the Stitch screens into a responsive Next.js frontend with mock data.
Do not connect the backend yet.
```

## Common Failure Modes

<Warning>
Do not assume AI Studio receiving a Stitch export means the app is production-ready. A generated prototype can still lack secure auth, database permissions, storage rules, API boundaries, and deployment hygiene.
</Warning>

<Warning>
Do not make one Stitch screen carry the entire product specification. A screen can imply data and flows, but it cannot reliably define row-level security, migrations, background jobs, or provider secrets.
</Warning>

<Warning>
Do not send a Stitch export directly into full-stack coding without first extracting an implementation brief. The agent may choose arbitrary data models or wire backend behavior around visual guesses.
</Warning>

<Tip>
Use `stitch-mcp-get-screen` when you need the raw screen details, `stitch-design-md` when you need a reusable visual system, `stitch-react-components` when you want React/Tailwind code, and `stitch-shadcn-ui` when you want accessible shadcn-style components.
</Tip>

## Final Checklist

- [ ] Decide whether the target is prototype, design handoff, or production engineering.
- [ ] Use Stitch for UI, flow, screenshots, HTML/CSS, and design theme.
- [ ] Use AI Studio for Google/Gemini-centered prototype work.
- [ ] Use Antigravity, Codex, Claude Code, Cursor, or Gemini CLI for engineering handoff.
- [ ] Retrieve screen data through Stitch MCP or Stitch skills when another agent needs to read the design.
- [ ] Generate `DESIGN.md` when the visual system should persist beyond one screen.
- [ ] Ask for an implementation brief before writing code.
- [ ] Build frontend with mock data before adding Supabase, Firebase, APIs, or auth.
- [ ] Treat backend architecture as a separate engineering decision, not as a visual export side effect.

## What To Remember

The relationship is subtle but not mystical:

```text
Stitch = design context
AI Studio = Google AI prototype and app-building surface
Antigravity/Codex/Claude Code/Cursor/Gemini CLI = agents that can consume the context
MCP/skills = the transport layer that makes Stitch readable outside Stitch
```

The best full-stack path is not to ask Stitch to become the backend. It is to let Stitch make the interface precise, then let a coding agent read that interface through MCP or export artifacts and build the application in clear phases.
