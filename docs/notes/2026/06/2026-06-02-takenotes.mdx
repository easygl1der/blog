## Summary

完整的 takenotes 流程可以理解成一套从零搭建的发布系统：先创建 Mintlify 知识库项目，再准备 `package.json`、`docs.json`、导出脚本和 Vercel 配置，然后把每次输入的原始素材生成 MDX，写入日期目录，更新导航，跑严格校验，提交到 GitHub，最后部署到 Vercel。零基础的人照着这篇做，目标不是只写一篇文章，而是做出一个可以反复发布知识笔记的网站。

---

## What This Solves

很多有价值的内容最开始都只是口头想法、聊天记录、部署过程、排错步骤或一段很乱的笔记。如果直接保存成普通文本，之后很难搜索、复用、发布，也很难保证格式正确。

Takenotes 解决的是这个问题：把原始材料转换成可发布、可导航、可校验、可部署的知识网页。它不是只写一篇文章，而是把文章放进一个完整发布系统里。

这条流水线做了几件事：

- 把一句话或一段聊天整理成教程、复盘、修复记录或研究笔记。
- 生成 Mintlify 可识别的 `.mdx` 文件。
- 自动放到按日期组织的目录里。
- 更新 `docs.json` 导航，让网页里能找到新文章。
- 运行 Mintlify 校验，提前发现格式、链接、可访问性问题。
- 用 Git 提交和推送，保留版本历史。
- 用 Vercel 部署，得到一个可以打开的 Preview URL。

## Who This Is For

这篇教程面向完全没跑过 takenotes 的人。你不需要先懂 MDX、Mintlify 或 Vercel 的所有细节，但需要能打开终端、知道项目文件夹在哪里，并愿意按步骤检查命令输出。

如果你已经会写 Markdown，这套流程会让你多理解三件事：文章如何进入站点导航、如何在提交前验证站点不会坏、如何把本地内容发布到公网。

## Prerequisites

- 一台能打开终端的电脑。
- Git，用来记录版本和推送到 GitHub。
- Node.js 和 npm，用来运行 Mintlify CLI、Vercel CLI 和构建脚本。
- 一个 GitHub 账号，用来存放博客仓库。
- 一个 Vercel 账号，用来把博客部署成网页。
- 一个本地博客目录，例如 `/Users/yitwah/projects/blog`。
- 一个 takenotes 技能或等价脚本，用来生成 MDX、写文件、更新导航和运行校验；如果没有技能，也可以按本文手动实现最小版本。

<Info>
在这个环境里，takenotes 的默认博客仓库是 `/Users/yitwah/projects/blog`，笔记默认写到类似 `docs/notes/2026/06/2026-06-02-example-slug.mdx` 的日期目录里，导航配置是 `/Users/yitwah/projects/blog/docs.json`。
</Info>

## The Workflow

<Steps>
  <Step title="安装最基础的本地工具">
    从零开始时，先确认电脑里能运行 Git、Node.js 和 npm。Git 负责提交版本，Node.js 负责运行 Mintlify 和 Vercel 的命令行工具。

```bash
git --version
node --version
npm --version
```

如果这些命令不存在，先安装 Git 和 Node.js。安装完成后重新打开终端再检查一次。
  </Step>
  <Step title="创建一个博客项目目录">
    选择一个固定目录作为知识库仓库。这个例子使用 `/Users/yitwah/projects/blog`，你也可以换成自己的路径。

```bash
mkdir -p /Users/yitwah/projects/blog
cd /Users/yitwah/projects/blog
git init
```

后面所有命令都默认在这个目录里执行。
  </Step>
  <Step title="创建最小目录结构">
    当前系统的关键目录很少：`docs/` 放 Mintlify 页面，`docs/notes/` 放按日期归档的笔记，`scripts/` 放构建脚本。

```bash
mkdir -p docs/notes/2026/06 scripts
```

一个可工作的最小结构大致是这样：

```text
blog/
  docs.json
  package.json
  vercel.json
  scripts/
    export-mintlify.mjs
  docs/
    index.mdx
    notes/
      2026/
        06/
          2026-06-02-takenotes.mdx
```
  </Step>
  <Step title="写 package.json">
    `package.json` 定义这个项目怎么安装依赖、怎么构建、怎么检查。当前项目使用 `mintlify` 生成站点，用 `adm-zip` 解压 Mintlify 导出的 zip。

```json
{
  "name": "yitwah-notes",
  "version": "0.1.0",
  "private": true,
  "type": "module",
  "scripts": {
    "build": "node scripts/export-mintlify.mjs",
    "vercel-build": "npm run build",
    "check": "mintlify validate && mintlify broken-links --check-anchors --check-redirects --check-snippets && mintlify a11y"
  },
  "devDependencies": {
    "adm-zip": "^0.5.16",
    "mintlify": "latest"
  }
}
```

写好后安装依赖：

```bash
npm install
```
  </Step>
  <Step title="写 Mintlify 导航配置 docs.json">
    `docs.json` 是站点的入口配置。它定义站点名字、主题颜色和导航。takenotes 发布新笔记时，最重要的动作之一就是把新页面路径加入这里。

```json
{
  "$schema": "https://mintlify.com/docs.json",
  "theme": "mint",
  "name": "Yitwah Notes",
  "colors": {
    "primary": "#2563eb",
    "light": "#60a5fa",
    "dark": "#0284c7"
  },
  "navigation": {
    "groups": [
      {
        "group": "Notes",
        "pages": [
          {
            "group": "2026",
            "pages": [
              {
                "group": "June",
                "pages": []
              }
            ]
          }
        ]
      }
    ]
  }
}
```

Mintlify 页面引用通常不写 `.mdx` 后缀，例如 `docs/notes/2026/06/2026-06-02-takenotes`。
  </Step>
  <Step title="写首页 docs/index.mdx">
    首页可以很简单，只要让 Mintlify 有一个入口页面即可。

```mdx
---
title: "Yitwah Notes"
description: "Personal knowledge notes and deployment guides."
---

# Yitwah Notes

This site collects practical notes, workflows, and technical guides.
```
  </Step>
  <Step title="写 Vercel 构建导出脚本">
    当前项目不是让 Vercel 直接运行一个长期服务器，而是用 Mintlify 导出静态站点，再把导出结果放进 `dist/`。脚本位置是 `scripts/export-mintlify.mjs`。

```js
import AdmZip from "adm-zip";
import { rmSync, mkdirSync } from "node:fs";
import { execFileSync } from "node:child_process";

const zipPath = "export.zip";
const outputDir = "dist";

rmSync(zipPath, { force: true });
rmSync(outputDir, { recursive: true, force: true });
mkdirSync(outputDir, { recursive: true });

execFileSync("npx", ["mintlify", "export", "--output", zipPath], {
  stdio: "inherit",
});

const zip = new AdmZip(zipPath);
zip.extractAllTo(outputDir, true);
```

这个脚本做四件事：删除旧导出文件，创建 `dist/`，运行 `mintlify export`，把 `export.zip` 解压给 Vercel 部署。
  </Step>
  <Step title="写 Vercel 配置">
    `vercel.json` 告诉 Vercel 最终部署哪个目录。因为构建脚本会把 Mintlify 导出结果放到 `dist/`，所以输出目录应该是 `dist`。

```json
{
  "outputDirectory": "dist"
}
```

然后本地先跑一次构建，确认 `dist/` 能生成：

```bash
npm run build
```
  </Step>
  <Step title="准备一段原始素材">
    到这里，网站骨架已经有了。每次 takenotes 的输入可以是一句话，例如“我是如何通过 Vercel 部署网页的”，也可以是一大段聊天记录、报错日志、命令输出或部署步骤。takenotes 会先判断这段素材适合变成哪种文章。

常见类型包括：

- `guide`：适合教程和完整流程。
- `error-fix`：适合错误症状、原因和修复。
- `devlog`：适合记录一次工程推进。
- `research`：适合整理调研结论。

如果目标是“让别人照着做”，通常选 `guide`。
  </Step>
  <Step title="把素材改写成结构化 MDX">
    一篇合格的 Mintlify MDX 需要 frontmatter、正文标题、摘要卡片、正文段落和元数据区域。frontmatter 是文件最开头的元信息，决定标题、描述、日期、类型、标签和发布状态。

```mdx
---
title: "通过 Vercel 部署网页"
description: "A beginner can deploy a website with Vercel by configuring the repo, build command, output directory, and environment variables."
date: "2026-06-02"
type: "guide"
tags: [vercel, deployment, frontend]
status: "published"
---

# 通过 Vercel 部署网页
```

对 `guide` 类型，正文至少应该包含这些部分：`What This Solves`、`Who This Is For`、`Prerequisites`、`The Workflow`、`Common Failure Modes`、`Final Checklist` 和 `What To Remember`。
  </Step>
  <Step title="理解一篇笔记的最终文件名">
    当前系统用日期加 slug 作为文件名。比如标题是“从零搭建 Takenotes 知识发布流程”，日期是 `2026-06-02`，最终文件可以是：

```text
docs/notes/2026/06/2026-06-02-takenotes.mdx
```

slug 要尽量短、英文、小写，用连字符连接。文件路径和导航引用要一致，导航引用不带 `.mdx`。
  </Step>
  <Step title="先写到临时 MDX 文件">
    实际发布前，可以先把生成的 MDX 放在临时文件里。这样即使校验失败，也不会污染正式笔记目录。

```bash
/tmp/takenotes-full-workflow-note.mdx
```

临时稿通过检查后，再交给发布脚本写入正式目录。
  </Step>
  <Step title="用发布脚本写入笔记目录">
    发布脚本会读取临时 MDX，检查 frontmatter 和正文结构，然后生成正式文件路径，并更新 Mintlify 导航。

```bash
python /Users/yitwah/.codex/skills/takenotes/scripts/publish_note.py \
  --vault /Users/yitwah/projects/blog \
  --date 2026-06-02 \
  --title "从零搭建 Takenotes 知识发布流程" \
  --type guide \
  --mdx /tmp/takenotes-full-workflow-note.mdx
```

成功后通常会得到一个类似这样的正式路径：

```text
/Users/yitwah/projects/blog/docs/notes/2026/06/2026-06-02-takenotes.mdx
```

这个脚本最重要的职责有五个：检查 MDX 是否有完整 frontmatter；检查 `guide` 类型是否包含必需章节；检查代码块是否都有语言标签；根据标题生成 slug 并写入日期目录；修改 `docs.json`，把新页面加入 `Notes -> 2026 -> June`。
  </Step>
  <Step title="检查 docs.json 导航">
    Mintlify 不是只看文件是否存在，还要看导航里有没有引用这个页面。`publish_note.py` 会把新页面加入 `docs.json` 的 `Notes -> 2026 -> June` 分组。

你应该确认新增的是本次这篇笔记，不要把无关草稿或未跟踪文件一起加入导航。

```bash
git diff -- docs.json
```
  </Step>
  <Step title="运行严格 Mintlify 校验">
    写入正式文件后，必须跑 Mintlify 校验。这个检查会验证构建、断链、锚点、可访问性和本地预览。

```bash
python /Users/yitwah/.codex/skills/takenotes/scripts/strict_mintlify_check.py \
  --config-dir /Users/yitwah/projects/blog
```

通过后，说明这篇笔记至少不会让 Mintlify 站点明显损坏。
  </Step>
  <Step title="用本地预览检查站点">
    严格检查会启动本地 Mintlify preview。你也可以手动启动，打开浏览器检查导航和文章排版。

```bash
npx --yes @mintlify/cli@latest dev --no-open
```

如果看到类似 `http://localhost:3000` 的地址，就说明本地站点已经跑起来。打开后确认新文章出现在左侧导航，正文没有明显排版错误。
  </Step>
  <Step title="检查 Git diff 和工作区">
    提交前要确认两件事：没有空白字符错误，以及暂存范围只包含本次任务需要的文件。

```bash
git diff --check
git status --short
git diff -- docs.json docs/notes/2026/06/2026-06-02-takenotes.mdx
```

如果工作区里有别人的修改或其他任务留下的文件，不要顺手提交。只提交本次笔记和必要的导航变更。
  </Step>
  <Step title="提交并推送到 GitHub">
    确认无误后，暂存本次文件并提交。

```bash
git add docs.json docs/notes/2026/06/2026-06-02-takenotes.mdx
git commit -m 'notes: add guide "从零搭建 Takenotes 知识发布流程" (2026-06-02)'
git push
```

提交信息要能说明文章类型、标题和日期，方便以后从 Git 历史里查找。
  </Step>
  <Step title="第一次连接 GitHub 仓库">
    如果这是一个全新项目，需要先在 GitHub 创建一个空仓库，再把本地仓库连接到远程。下面的地址只是示例，要替换成你自己的 GitHub 仓库地址。

```bash
git remote add origin https://github.com/your-name/blog.git
git branch -M main
git push -u origin main
```

之后每次发布新笔记，只需要普通 `git push`。
  </Step>
  <Step title="用 Vercel 部署 Preview">
    第一次部署前，先确认 Vercel CLI 能识别你的账号。如果没有登录，CLI 会提示你登录。

```bash
npx --yes vercel whoami
```

然后从博客目录发起 preview 部署：

```bash
npx --yes vercel deploy /Users/yitwah/projects/blog -y
```

部署完成后，Vercel 会输出 Preview URL 和 READY 状态。Preview 用来检查页面是否能打开、导航是否出现新文章、内容是否排版正常。
  </Step>
  <Step title="需要正式上线时再发 Production">
    Preview 没问题后，才考虑生产发布。生产部署要显式加 `--prod`。

```bash
npx --yes vercel deploy /Users/yitwah/projects/blog --prod -y
```

如果只是记录和测试，preview 已经足够；如果要让正式域名访问最新内容，再发布 production。
  </Step>
  <Step title="形成以后每次复用的固定命令">
    搭建完成后，每次发布新 takenotes 都是同一套顺序。

```bash
python /Users/yitwah/.codex/skills/takenotes/scripts/publish_note.py \
  --vault /Users/yitwah/projects/blog \
  --date 2026-06-02 \
  --title "新的笔记标题" \
  --type guide \
  --mdx /tmp/new-note.mdx

python /Users/yitwah/.codex/skills/takenotes/scripts/strict_mintlify_check.py \
  --config-dir /Users/yitwah/projects/blog

git diff --check
git status --short
git add docs.json docs/notes/2026/06/2026-06-02-new-note.mdx
git commit -m 'notes: add guide "新的笔记标题" (2026-06-02)'
git push
npx --yes vercel deploy /Users/yitwah/projects/blog -y
```

真正需要你每次认真看的不是命令本身，而是 `git status --short` 和 `git diff -- docs.json`。它们决定这次提交是否只包含你想发布的内容。
  </Step>
</Steps>

<Tip>
Takenotes 最重要的习惯是每次只提交本次笔记和对应导航。写文章、校验、提交、部署都可以自动化，但提交范围必须人工确认。
</Tip>

## Common Failure Modes

<Warning>
中文 description 可能被简单校验器误判为“词数不足”。如果发布脚本按空格分词，可以在 description 中使用一条清晰英文句子，或者写成带空格的中英混合描述。
</Warning>

<Warning>
MDX 代码块必须带语言标签。不要写空的三反引号，应该写成 `bash`、`text`、`mdx`、`json` 等明确类型。
</Warning>

<Warning>
发布脚本更新导航时，可能把同日期目录里的其他未跟踪笔记也加入 `docs.json`。提交前一定要看 `git diff -- docs.json`，只保留本次要发布的页面引用。
</Warning>

<Warning>
不要在未确认的情况下用 `git reset`、`git restore`、`git clean` 或删除文件来修正状态。修正 Git 状态可能影响用户已有改动，必须先确认要改什么。
</Warning>

<Warning>
Vercel 构建时出现 npm audit 警告不一定代表部署失败。真正要看的是构建命令是否完成、输出是否部署成功，以及最终 readyState 是否为 READY。
</Warning>

## Final Checklist

- [ ] 原始素材已经明确要写成哪种文章，通常是 `guide`。
- [ ] MDX frontmatter 包含 `title`、`description`、`date`、`type`、`tags` 和 `status`。
- [ ] 正文有 H1 标题，并包含 guide 必需章节。
- [ ] 所有代码块都有语言标签。
- [ ] `publish_note.py` 成功写入正式笔记路径。
- [ ] `docs.json` 只新增本次页面引用。
- [ ] `strict_mintlify_check.py` 通过。
- [ ] `git diff --check` 通过。
- [ ] commit 只包含本次笔记和导航变更。
- [ ] `git push` 成功。
- [ ] Vercel preview 部署完成并显示 READY。

## What To Remember

Takenotes 不是简单把聊天内容复制成 Markdown，而是一个可重复的发布系统。输入是原始想法，输出是一篇通过校验、进入导航、写入 Git 历史、部署到 Vercel 的网页。只要每次都按“生成 MDX -> 发布脚本 -> 校验 -> 检查 diff -> 提交推送 -> preview 部署”的顺序走，零基础也能稳定发布自己的知识库。
